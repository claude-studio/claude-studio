# Summary

Claude Studio should not rely on a settings-only language control. The strongest evidence across W3C, NN/g, USWDS, and UK government design guidance points to a visible language switch in the app shell, available on every page, with a secondary settings path for management and explanation. For a desktop-first shell like Claude Studio, the best fit is a persistent top-bar action on the right side, plus a mirrored control in `/data` as the backup path. W3C explicitly warns that automatic language detection is only a starting point and that users need an obvious override wherever they are in the product ([W3C Accept-Language](https://www.w3.org/International/questions/qa-accept-lang-locales.en.html), [W3C language negotiation](https://www.w3.org/International/questions/qa-when-lang-neg.en.html), [W3C translated pages](https://www.w3.org/International/questions/qa-site-conneg.en.html)).

For only two supported languages, a direct toggle is more discoverable than a generic dropdown. The visible label should use autonyms, not flags: show `한국어` when the UI is in English, and `English` when the UI is in Korean, with a secondary full control in settings listing `English` and `한국어`. This matches government two-language patterns that show the target language directly and accessibility guidance that each option should be marked with its own `lang` value ([DWP language toggle](https://design-system.dwp.gov.uk/components/language-toggle), [DWP accessibility](https://design-system.dwp.gov.uk/components/language-toggle/accessibility), [ONS change language](https://service-manual.ons.gov.uk/design-system/patterns/change-language), [MDN `lang`](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/lang)).

Because the product requirement is English-default, Claude Studio should keep English as the first-launch fallback, but that choice increases the burden on discoverability. The app should therefore pair the default with a highly visible shell-level switch and, when the OS/browser language is Korean and no user choice exists yet, a one-time non-modal hint pointing to `한국어`. After the user explicitly chooses a language, that stored preference should override OS/browser signals until changed again ([W3C Accept-Language](https://www.w3.org/International/questions/qa-accept-lang-locales.en.html), [W3C translated pages](https://www.w3.org/International/questions/qa-site-conneg.en.html), [NN/g](https://www.nngroup.com/articles/language-switching-ecommerce/)).

# Placement Guidance

- Put the primary language control in the persistent app shell, not only in `/data`. W3C recommends language controls on all pages because automatic negotiation can fail and users may land deep-linked into the product ([W3C language negotiation](https://www.w3.org/International/questions/qa-when-lang-neg.en.html), [NN/g international web usability](https://www.nngroup.com/articles/international-web-usability/)).
- For Claude Studio specifically, place the control in the top bar at the far right. This best matches the cross-source guidance to use the upper corners or header/service navigation for discoverability ([NN/g language switching](https://www.nngroup.com/articles/language-switching-ecommerce/), [USWDS two languages](https://designsystem.digital.gov/patterns/select-a-language/two-languages/), [GOV.UK navigate a service](https://design-system.service.gov.uk/patterns/navigate-a-service/)).
- Keep a second path in `/data` under a clearly named `Language` section. The shell action is for immediate recovery; settings is for confirmation, explanation, and persistence details. This matches the requirement for both discoverability and a backup access path.
- Do not make the sidebar footer the only visible entry. Footer placement is repeatedly called out as weak for discoverability, especially on smaller or dense interfaces ([NN/g language switching](https://www.nngroup.com/articles/language-switching-ecommerce/)).
- Do not use a first-run modal as the only path. W3C and NN/g support in-flow switching and visible controls; forcing everyone through a language gate is described as poor human-factors design when a default page can load first ([W3C language negotiation](https://www.w3.org/International/questions/qa-when-lang-neg.en.html), [NN/g international web usability](https://www.nngroup.com/articles/international-web-usability/)).
- For the public landing page, use the same pattern: top-right header placement on desktop, above the fold or inside primary nav on mobile, with the same label treatment and persistence behavior ([NN/g language switching](https://www.nngroup.com/articles/language-switching-ecommerce/), [USWDS two languages](https://designsystem.digital.gov/patterns/select-a-language/two-languages/)).

# First-Run and Persistence Guidance

- On first launch with no saved preference, English may remain the product default because that is a stated business requirement. This is a product decision, not the default UX recommendation from the sources; the compensating requirement is a highly visible override.
- Treat OS/browser language as a hint, not truth. W3C explicitly says `Accept-Language` is acceptable for first contact only and must not lock the user in ([W3C Accept-Language](https://www.w3.org/International/questions/qa-accept-lang-locales.en.html)).
- If the OS/browser language is Korean and there is no saved in-app choice, show a one-time non-modal prompt anchored to the shell control, for example: `한국어를 찾고 계신가요? 상단의 한국어 버튼으로 변경할 수 있습니다.` This is an inference from the evidence: the sources support visible override, sticky user choice, and avoiding forced modals; the hint is the lowest-friction way to satisfy the English-default constraint without trapping Korean-first users.
- After explicit user choice, persist that locale locally and always prefer it over OS/browser signals. W3C recommends remembering the chosen language and reusing it on later visits ([W3C translated pages](https://www.w3.org/International/questions/qa-site-conneg.en.html), [W3C language negotiation](https://www.w3.org/International/questions/qa-when-lang-neg.en.html)).
- Keep the control in the same location after switching. Consistent placement matters more than relabeling cleverness; users who learned where the switch lives should not have to rediscover it ([USWDS two languages](https://designsystem.digital.gov/patterns/select-a-language/two-languages/), [NN/g language switching](https://www.nngroup.com/articles/language-switching-ecommerce/)).
- For web, persist the same way and do not auto-redirect based on location. USWDS explicitly warns against auto-redirect by location or browser settings alone ([USWDS two languages](https://designsystem.digital.gov/patterns/select-a-language/two-languages/)).

# Copy and Accessibility Rules

- Use autonyms/endonyms for language names: `English`, `한국어`. Multiple sources recommend native language names because they remain legible to people who cannot read the current UI language ([NN/g international web usability](https://www.nngroup.com/articles/international-web-usability/), [NN/g language switching](https://www.nngroup.com/articles/language-switching-ecommerce/), [MDN `lang`](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/lang)).
- Do not use flags to represent languages. W3C and USWDS both reject that pattern because countries and languages do not map cleanly ([W3C specifying language](https://www.w3.org/International/geo/html-tech/tech-lang.html), [USWDS two languages](https://designsystem.digital.gov/patterns/select-a-language/two-languages/)).
- For the shell-level two-language control, prefer a direct target-language action over a generic dropdown. In English UI, show `한국어`; in Korean UI, show `English`. In settings, use a fuller label such as `Language` with radio options `English` and `한국어`. This follows the two-language government toggle pattern and improves immediate recovery for Korean-first users ([DWP language toggle](https://design-system.dwp.gov.uk/components/language-toggle), [ONS change language](https://service-manual.ons.gov.uk/design-system/patterns/change-language)).
- Add a screen-reader-only verb phrase for the direct toggle, for example `Change language to 한국어` or `언어를 English로 변경`. ONS uses exactly this pattern with visually hidden prefix text so the visible UI stays compact while assistive tech gets a full action label ([ONS change language](https://service-manual.ons.gov.uk/design-system/patterns/change-language)).
- Mark each language label with its own language metadata: `lang="en"` for `English`, `lang="ko"` for `한국어`. Mixed-language labels otherwise may be pronounced incorrectly by screen readers ([WCAG language of parts](https://www.w3.org/WAI/WCAG22/Understanding/language-of-parts.html), [MDN `lang`](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/lang), [WebAIM language](https://webaim.org/techniques/language/)).
- Update the root document language on switch: web should update `<html lang="en">` or `<html lang="ko">`; Electron-rendered content should do the same. This is required for correct pronunciation and language detection ([MDN `lang`](https://developer.mozilla.org/en-US/docs/Web/HTML/Reference/Global_attributes/lang), [WebAIM language](https://webaim.org/techniques/language/)).
- Preserve logical focus after switching. If the trigger remains mounted, return focus to the trigger after the menu closes or after rerender. WCAG focus-order guidance favors returning focus to the invoking control when the temporary UI closes ([WCAG focus order](https://www.w3.org/WAI/WCAG22/Understanding/focus-order.html)).
- Announce the result of a locale change with a polite live region such as `Language changed to 한국어` or `언어가 English로 변경되었습니다.` Use `role="status"` and `aria-atomic="true"` so the change is announced without stealing focus ([ARIA22](https://www.w3.org/WAI/WCAG22/Techniques/aria/ARIA22)).
- Design for string growth and height changes. W3C notes that short UI strings can expand by 200% to 300% in translation, and CJK scripts can require more width or line height even when character count is similar ([W3C text size in translation](https://www.w3.org/International/articles/article-text-size.en.html)).
- Avoid fixed-width buttons, tabs, badges, and top-bar slots that assume current Korean labels will stay short after English conversion. Let labels wrap where possible, and prefer min-width plus padding over hard width.
- Avoid abbreviations and transliterated pseudo-English where a natural product term exists. W3C warns that abbreviations often do not survive translation well, and short English labels are exactly where expansion risk is highest ([W3C text size in translation](https://www.w3.org/International/articles/article-text-size.en.html)).
- Use one language per screen by default. Do not turn ordinary navigation and status text into bilingual doubles such as `Projects / 프로젝트` across the product. Reserve bilingual presentation for language selection, onboarding/recovery hints, and support surfaces where discoverability is the goal. This is an inference from the sources: they consistently recommend native labels for switching and visible controls, not permanently duplicated UI copy.
- Keep English and Korean pairs semantically equivalent, not word-for-word. Product copy should preserve intent, action, and tone across languages rather than mirror syntax.
- Navigation labels should stay short nouns. Buttons should use direct verbs. Empty states should use a single plain sentence plus one clear next action. Status text should be brief and descriptive, not idiomatic or playful. This aligns with the plain-language bias in government design guidance and reduces localization ambiguity.
- Do not rely on color or motion alone to signal that language changed. The visible state must update textually, and the announcement must be available to assistive technologies ([ARIA22](https://www.w3.org/WAI/WCAG22/Techniques/aria/ARIA22)).

# Claude Studio Recommendation

- Primary placement: add a persistent top-bar language action at the far right of the existing title bar. Because Claude Studio only needs English and Korean today, make it a direct toggle, not a buried submenu.
- Visible labeling: on English screens, show a globe icon plus `한국어`; on Korean screens, show a globe icon plus `English`. Keep the visible label in the target language so a Korean-only user can recover immediately.
- Accessible naming: give the control a fuller accessible name, for example `Current language English. Change language to 한국어.` and localize that string in both directions. Mark the visible target-language token with the appropriate `lang`.
- Backup path: add a `Language` section near theme controls on `/data` with a short explanation, radio options `English` and `한국어`, and note that the choice applies across the app and landing page when possible.
- First launch: default to English if no stored preference exists. If the OS/browser language is Korean and no explicit choice exists yet, show a one-time non-modal hint pointing to the top-bar `한국어` action. Do not block the app behind a modal picker.
- Persistence: once the user chooses a language, persist it locally and let that override OS/browser language until the user changes it.
- Web landing page: mirror the same header placement and label behavior so the web and desktop entry points teach the same interaction.
- Future-proofing: if Claude Studio adds a third language, replace the direct toggle with a header dropdown labeled `Language` and list options as autonyms (`English`, `한국어`, etc.), each with its own `lang`.

# Gaps

- I did not find a strong primary-source style guide specific to English/Korean product microcopy tone, so the copy rules above are derived from multilingual UX, accessibility, and localization standards rather than a Korean product-writing authority.
- Material Design’s language-support page was too general to add decision-grade guidance for switcher placement, so it is not part of the recommendation set.
- Apple HIG did not surface a comparably specific, citable language-switcher pattern in the research pass. If the team wants platform-native macOS wording or restart-behavior guidance, that should be a separate targeted pass.
- The recommendation to keep English as first-launch default while showing a Korean hint is an implementation inference driven by the product requirement. The standards sources would ordinarily support starting from user language preferences when allowed.

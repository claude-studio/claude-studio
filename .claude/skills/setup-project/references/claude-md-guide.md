# CLAUDE.md 작성 가이드

## 핵심 원칙

- **200줄 이하** 유지 — 초과 시 rules/ 파일로 분리
- **항상 로드됨** — 모든 컨텍스트에서 읽힌다, 군더더기 금지
- 핵심 지시문만 직접 기술 (상세 규칙은 `.claude/rules/`에 작성)
- 플레이스홀더는 실제 내용으로 반드시 교체

## 4개 필수 섹션

1. **프로젝트 설명** — 한 줄 소개 + 기술 스택
2. **핵심 명령어** — 개발/빌드/테스트 명령어
3. **아키텍처** — 주요 디렉터리/모듈 구조
4. **주의사항** — 절대 하지 말아야 할 것들

## CLAUDE.md 플레이스홀더 템플릿

```markdown
# <!-- 프로젝트 이름 -->

<!-- 한 줄 프로젝트 설명. 예: "사용자 인증 서비스. Express + PostgreSQL." -->

## 핵심 명령어

\`\`\`bash

# 개발 서버

<!-- 예: npm run dev / pnpm dev / python manage.py runserver -->

# 빌드

<!-- 예: npm run build / pnpm build -->

# 테스트

<!-- 예: npm test / pytest -->

# 린트

<!-- 예: npm run lint -->

\`\`\`

## 아키텍처

<!-- 주요 디렉터리/모듈과 역할을 짧게 나열. 예:
- `src/` → 소스코드
- `src/api/` → REST API 핸들러
- `src/db/` → 데이터베이스 모델
-->

## 주의사항

<!-- 절대 하지 말아야 할 것들. 예:
- 시크릿/API 키 하드코딩 금지 → 반드시 환경변수 사용
- `main` 브랜치 직접 푸시 금지
-->
```

## 분리 기준

CLAUDE.md에 직접 쓰지 말고 rules/로 분리해야 하는 항목:

| 항목                      | 분리 방법                                 |
| ------------------------- | ----------------------------------------- |
| Git 커밋/브랜치 규칙 상세 | `rules/git.md`                            |
| 모노레포 패키지 구조      | `rules/monorepo.md`                       |
| API 설계 규칙             | `rules/api.md` (paths: `src/api/**`)      |
| 테스트 작성 규칙          | `rules/testing.md` (paths: `**/*.test.*`) |
| 특정 기능 구현 패턴       | `rules/{feature}.md` (paths 지정)         |

## 크기 모니터링

- 150줄 초과 → 규칙 파일 분리 검토
- 200줄 초과 → 반드시 분리 (200줄 이후 잘림)
- 분리 후 CLAUDE.md에는 "자세한 내용은 `.claude/rules/git.md` 참조" 한 줄만 남기기
